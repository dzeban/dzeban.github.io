Yii framework. Модель.

27 фев 2011, в категории IT

Теория

Продолжаю писать про Yii. Сегодня поговорим о модели. Поговорим спокойно, обстоятельно и не торопясь.

Итак, согласно документации по Yii, модель «используется для хранения данных и применимых к ним бизнес-правил».

Вспоминаем великую идею MVC о разделении кода приложения на 3 части — управление данными, внешний вид, взаимодействие предыдущих двух.

Теперь нужно очень четко понять суть моделей.

Так вот, модель — это НЕ механизм доступа к данным. Очень многие так считают, однако понятие модели намного шире и оно становится предельно ясным, если немного абстрагироваться.

Забудьте о PHP, забудьте о том, что вы делаете сайт. Вы делаете приложение. Ваше приложение работает с некими сущностями, которые вам дают «что-то», а вы с этим «что-то» чего-то делаете. В качестве сущности может быть что угодно — карта мира, двигатель автомобиля, матрица. Подчеркиваю — приложение работает с сущностями.

Так вот, модель вашего приложения это механизм работы с сущностями, который содержит в себе всё для работы с этими сущностями — правила, ограничения, поведение, доступ к этим сущностям.

Приведу аналогию из реальной жизни:

Вы пишете приложение «оркестр».

  • Сущность — музыкальный инструмент
  • Модель — музыкант
  • Контроллер — дирижер
  • Вид(представление) — уши слушателей

Музыкант(модель) знает всё о работе с музыкальным инструментом(сущностью) — как его держать (доступ к сущности), как на нем можно играть(правила), как на нем нельзя играть(ограничения). И он на нем, собственно, играет(поведение).

Но музыкант не будет играть на инструменте, пока ему не скажет дирижер(контроллер). Дирижер(контролер) знает как сказать музыканту(модели), что делать (использует интерфейс модели). В результате мы получаем музыку через наши уши(представление).

Надеюсь, стало понятно.

Каждая часть MVC — это кучка PHP-кода. И на самом-то деле, MVC-кодинг не обязывает нас использовать какой-либо фреймворк. MVC — это паттерн программирования, т.е. подход к архитектуре ПО. Мы можем сами писать в MVC-стиле, выделив 3 части и сделав для каждой из них интерфейс, используя который, другие части могут с ней взаимодействовать.

MVC-фреймворк же предоставляет уже готовые куски базового кода для того, чтобы кодить в этом паттерне. Этот базовый код предоставляет реализованные интерфейсы, служебные функции, какие-то обработчики ошибок и прочие вещи, которые пришлось бы писать каждый раз.

Практика

Всё, что я сказал выше, на практике представляется следующим образом (для ООП, естесственно):

  • Сущности представляются в виде классов
  • Атрибуты сущности — свойства класса
  • Доступ к сущности, ограничения и пр. — методы класса

Каждый фреймворк реализует это по-своему и мы рассмотрим вариант Yii. Мы не будем следовать официальному туториалу Yii и будем изучать модель с самого низа. О чем это я? Читая документацию можно подумать, что в Yii всего 2 сущности — запись БД и форма. Это, на самом деле, не совсем правда — это те сущности, для которых уже реализованы модели. В Yii они представляются как CFormModel и CActiveRecord — модели для работы с формами и БД соответственно. Это уже готовые модели, которые можно просто брать и использовать.

На низком же уровне модель в Yii представляется в виде класса CModel, который содержит в себе реализацию основных вещей и, наследуя от которого, мы создаем собственные модели и начнем мы с чего-нибудь примитивного.

Вообще, работу с примитивными объектами, например plain text, можно сделать средствами чистого PHP, передавая их через метод render() в контроллере. И в реальной жизни бессмысленно заводить под такие данные модель. И вообще, не нужно забывать зачем нужны все эти фреймворки — для того, чтобы писать большие, сложные приложения быстро и просто. А когда используют обычный текст для хранения информации — это несерьезно и вам тогда вообще не нужен никакой фреймворк.

Давайте представим, что мы пишем приложение RSS-агрегатор. В нем есть, например, такая сущность как подписка (feed).

Модель подписок

Создаем модель —делаем файл protected/models/Feed.php

В нем пишем

<?php
class Feed extends CModel
{
};
?>

Это заготовка для нашей модели. Как видно, мы отнаследовали наш класс модели от класса CModel. Опишем атрибуты сущности и способы работы с ними.

У подписки есть следующие атрибуты:

  • Название
  • Описание
  • Автор
  • Ссылка

Как я уже говорил раньше, атрибуты сущности задаются в модели через свойства класса, а все механизмы работы через методы. Поэтому то, что я описал выше представляется примерно вот так:

<?php
class Feed extends CModel
{
    public $title;
    public $description;
    public $author;
    public $link;
}
?>

Если у вас такая модель как сверху, то код не будет работать. Это происходит потому что в классе CModel есть абстрактный метод attributeNames, который обязательно нужно переопределить иначе возникнет ошибка вида.

Fatal error: Class Feed contains 1 abstract method and
must therefore be declared abstract or implement the
remaining methods.

Поэтому добавляем код для этого метода, который собственно и будет возвращать эти имена

<?php
public function attributeNames()
{
    return array(
            'title',
            'description',
            'author',
            'link'
            );
}

Чтобы это заработало надо добавить в конфиг автозагрузку моделей. Для этого просто добавляем в protected/config/main.php

<?php
return array(
        'basePath'=>dirname(__FILE__).DIRECTORY_SEPARATOR.'..',
        'name'=>'MVC models & Yii',
        'import'=>array(
            'application.models.*'
            )
        );
?>

Теперь можно в контроллере protected/controllers/SiteController.php создавать экземпляр класса:

<?php
class SiteController extends CController
{
    function actionIndex()
    {
        $feed = new Feed();
    }
}

Хорошо, атрибуты мы задавать научились. Теперь хотелось бы уже сделать чего-то полезного. Для начала определимся чего мы хотим. На первых порах от модели хотелось бы следующего:

  1. Чтобы она задавала значения своих атрибутов
  2. Чтобы она отдавала значения своих атрибутов
  3. Чтобы она проверяла корректность своих атрибутов

Первые два пункта делаются с помощью методов setAttributes() и getAttributes() соответственно. Эти методы уже реализованы в CModel, т.е. их нам писать не неадо. Они просты как молоток, и про них можно посмотреть на сайте Yii. Вкратце, им передается массив с именами аттрибутов которые хотим взять/задать. Однако, чего нет на сайте Yii так это вот такого маленького момента, из-за которого я чуть с ума не сошел. Если наши аттрибуты объявить как private, то setAttributes их не увидит и будет ругаться словами Property "Feed.title" is not defined. Если же аттрибуты public или даже protected, то всё нормално. С чем это связано — я не понял.

В конечно итоге, я написал в контроллере такой нехитрый код для работы с атрибутами

<?php
$feed->setAttributes(
        array(
            'title'=>'Feed title',
            'description'=>'Feed description',
            'author'=>'Feed author',
            'link'=>'Feed link'
            ),
        false
        );
var_dump($feed->getAttributes());

Второй аргумент в вызове setAttributes это параметр $safeOnly. По умолчанию он установлен в true, а значит будет устанавливать значения только для тех аттрибутов, которые прошли проверку на валидность (об этом дальше).

Собственно, это все хитрости связанные с изменением аттрибутов.

Другое дело — проверка корректности. Для этого в Yii используются так называемые валидаторы. Это такие методы, которые получают на вход аттрибуты и в случае ошибки вызывают метод addError(). В качестве валидатора можно использовать какой-либо метод класса, создать свой собственный валидатор, отнаследовав его от CValidator, либо же воспользоваться одним из уже готовых валидаторов в поставке Yii.

Каждому атрибуту можно назначить один или несколько валидаторов. Задаются они с помощью метода rules() в классе нашей модели.

<?php
public function rules()
{
    return array(
            array('title, link', 'required'),
            array('link', 'url'),
            );
}

Видно, что правила организованы в массив. Каждое правило также представляет собой массив, который имеет формат.

'список аттрибутов', 'применяемый валидатор', 'аргументы валидатора', 'on'=>'сценарий валидации'

В моем случае я попросил, чтобы название и ссылка подписки были не пусты, а также чтобы ссылка на подписку была составлена правильно.

Чтобы проверить на валидность вызывается метод validate(), ошибки от которого можно проверить с помощью метода getErorrs(). Выглядит это примерно так:

<?php
if(!$feed->validate())
{
    var_dump($feed->getErrors());
}

Это делается, естесственно, в контроллере.

В моем случае getErrors мне вернул ошибку

<?php
array(1) {
    ["link"]=>
        array(1) {
            [0]=>
                string(24) "Link is not a valid URL."
        }
}

то есть атрибут link не валидный.

Проверку валидаторами можно организовать в т.н. сценарии. Сценарий — это определенная последовательность валидаторов. Например, могут быть разные сценарии валидации при логине и при регистрации пользователя. В последнем случае помимо полей имени пользователя и пароля  обязательным является поле e-mail.

Сценарий — четвертый ‘on’-аргумент в массиве правил валидации. Для того, чтобы указать по какому сценарию провести валидацию задаётся свойство scenario класса CModel. В будущих статьях я, возможно,  это подробнее расскажу.

Полный код модели, который даже что-то делает:

<?php

class Feed extends CModel
{
    public $title;
    public $description;
    public $author;
    public $link;

    public function attributeNames()
    {
        return array(
                'title',
                'description',
                'author' ,
                'link'
                );
    }

    public function attributeLabels()
    {
        return array(
                'title' => 'Title',
                'description' => 'Description',
                'author' => 'Author' ,
                'link' => 'Link'
                );
    }

    public function rules()
    {
        return array(
                array('title, link', 'required'),
                array('link', 'url'),
                );
    }
}

И контроллер

<?php
class SiteController extends CController
{

    function actionIndex()
    {
        $feed = new Feed();
        echo '<pre>';
        var_dump($feed->attributeNames());

        $feed->setAttributes(
                array(
                    'title'=>'Feed title',
                    'description'=>'Feed description',
                    'author'=>'Feed author',
                    'link'=>'Feed link'
                    ),
                false
                );
        var_dump($feed->getAttributes());

        if(!$feed->validate())
        {
            var_dump($feed->getErrors());
        }

        echo '</pre>';
    }
}

?>

Итог

Вообще говоря, по моделям сказать больше нечего. По крайней мере на том уровне, на котором я только что рассказал. Еще раз напомню, что CModel — это очень низкоуровневый класс, который дает только следующие вещи:

  • Установить/забрать атрибуты
  • Сделать валидацию
  • Функции обработки ошибок

И ничего более. Поэтому создавать какое-то реально полезное приложение на таком низком уровне просто глупо. Для работы с базами данных и прочими полезными штуками расскажу дальше.

По поводу обработки ошибок хочется сказать еще пару слов. Для того, чтобы сообщения об ошибках были красивые и понятные можно задать имена аттрибутов (attribute’s label). Я не стал передовить label как “метка”, потому что это убого. Так вот, например, чтобы вместо

Error with $someVariable

писать

Error with current feed list

мы задаем эти самые имена. Задаются они с помощью метода attributeLabels(). Если посмотреть в исходники Yii, то там видно, что по умолчанию этот метод возвращает пустой массив. А вот мы его возьмем и переопределим

<?php
public function attributeLabels()
{
    return array(
            'title' => 'Title',
            'description' => 'Description',
            'author' => 'Author' ,
            'link' => 'Link'
            );
}

В Yii есть дефолтная функция для генерации меток. Она заменяет подчеркивания и дефисы на пробелы, делает первые буквы заглавными. Эта функция состоит всего лишь из одной строчки, но зато какой

<?php
return ucwords(trim(strtolower(str_replace(array('-','_','.'),' ',preg_replace('/(?<![A-Z])[A-Z]/', ' \0', $name)))));

В большинстве случаев этого вполне хватит, но если хотите экзотики, то attributeLabels вам в помощь.
